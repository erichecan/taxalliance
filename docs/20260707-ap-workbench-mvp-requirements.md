# AP 工作台 — 第一步需求辨析

> 日期：2026-07-07
> 背景：与合伙人讨论后决定「先解决第一步」——客户资料可扫描、可识别、自动录入记账软件（QuickBooks）。
> 本文档是需求讨论的结论沉淀，是后续 PRD / 技术方案 / DEV-PLAN 的锚。
> 与 `20260622-product-plan.md`、`20260702-plan-supplement.md` 的关系：本文档是那两份大规划的**第一步落地聚焦**，冲突处以本文档为准。

## 一、这次要解决的真问题

中小企业每月有一堆供应商发票/收据，会计师（记账公司）要一张张手工录入客户的记账软件、分类科目、走审批、安排付款、再归档。**这套应付账款（AP）处理又慢又容易错**。

第一步要做的，就是把「会计师帮一批中小企业客户处理应付账款」这件事，从**手工**变成**AI 识别 + 人工复核 + 一键录入**。

## 二、产品定位（一句话）

**给会计师/记账公司的「多客户应付账款（AP）工作台」**：会计师在一个后台里，帮 N 个中小企业客户完成「收单 → OCR 识别 → 自动分类 → 录入客户 QuickBooks → 复核/审批 → 付款指令 → 归档」。

## 三、已定决策（2026-07-07 讨论拍板）

| # | 决策点 | 定论 |
|---|--------|------|
| 1 | 主用户 / 买家 | **会计师 / 记账公司**（不是企业财务部自己）；产品是**多租户、多客户工作台** |
| 2 | 范围形态 | 完整 AP：收单 → OCR → 分类 → 录入 QBO → 复核+审批 → **付款指令** → 归档 |
| 3 | 付款落地 | **不碰钱**：只生成付款指令/批次（或在 QBO 标记 Bill 待付），实际转账由企业在自己网银 / QBO Bill Pay 执行 → 规避支付牌照与信托合规，最快上线 |
| 4 | 差异化立足点 | ① 华人/多语种 + 会计师联盟渠道分销；② AP 数据直通 T2 报税；③ 人才基地人工兜底（100% 结果经人复核） |
| 5 | 邮件入口 | 每个客户企业分配一个**专属转发邮箱**（像 Dext/Hubdoc），客户把发票发过来；会计师也能手动上传。**不绑死 Outlook** |
| 6 | 审批 | 两级：① 会计师复核**记账分类**（质量关）；② 企业老板授权**付款**（钱的关） |
| 7 | 归档 | 系统内统一文档库，可选回传客户 QBO 附件 / SharePoint / Google Drive。**不绑死 SharePoint** |
| 8 | 记账软件 | 第一步只接 **QuickBooks Online**（加拿大市占最高）；Xero 后续 |
| 9 | **MVP 边界** | 只做**核心链**：收单 → OCR → 分类 → 复核 → 录入 QBO（**单客户跑通**）。审批/付款/多客户/报税/人工兜底都在核心链验证后再叠加 |

## 四、MVP 范围：做什么（核心链，单客户）

1. **收单**：专属转发邮箱收取发票/收据（PDF、图片、扫描件）+ 手动上传
2. **OCR 识别**：调 Veryfi API，结构化提取（供应商、日期、金额、税额 GST/HST、行项目），返回置信度
3. **AI 自动分类**：把每条交易映射到 GL 科目（Claude API + 规则引擎混合），置信度分级（高自动通过 / 中建议 / 低必须人工）
4. **会计师复核工作台**：左原件预览 | 中 AI 识别结果 | 右目标科目；逐条复核、内联编辑、置信度高亮、异常提示
5. **一键录入 QBO**：复核确认后写入 QuickBooks Online（沙箱，创建 Bill/Expense），原件作附件回传，按「日期+金额+供应商」去重

## 五、明确不做（第一步砍掉 / 推迟）

- ❌ 审批流、付款指令、付款执行（核心链跑通后再叠）
- ❌ 多客户 / 多租户权限界面（**但数据库 schema 从一开始带 `client_id`，预留多租户，避免返工**）
- ❌ T2 报税打通（GIFI 映射、附表预填）
- ❌ 人才基地人工兜底工单系统对接
- ❌ 归档回传 SharePoint / Google Drive
- ❌ Xero / 其他记账软件
- ❌ 加盟品牌体系（方向 B）
- ⏸️ 多语种/中文发票专项优化：紧接核心链之后，MVP 先跑通标准英文发票

## 六、完整数据流（MVP 版）与用户原流程图的对应

```
客户发邮件 ──▶ 专属转发邮箱（非绑定 Outlook）
                    │
              Veryfi OCR 识别（金额/日期/供应商/税/行项目）
                    │
              AI 分类到 GL 科目（Claude + 规则，带置信度）
                    │
              会计师复核工作台（逐条确认/修正）
                    │
              一键录入 QuickBooks Online（Bill/Expense + 附件 + 去重）
                    ┈┈┈ 以下为第一步之后叠加 ┈┈┈
              经理审批 → 付款指令 → 归档文档库 → 报税打通
```

## 七、边界、外部依赖与坑

| 项 | 说明 |
|----|------|
| QuickBooks API | 免费，无按量计费。但**生产密钥需过 Intuit 安全审查（约 2–4 周）**。MVP 用沙箱开发，并行提交审查申请 |
| Veryfi OCR | 开发期用**永久免费 100 份/月**额度足够；上线放量再评估 Starter（$500/月最低消费） |
| 多租户预留 | MVP 只做单客户界面，但 schema 一开始就带客户维度，避免后期大改 |
| 分类学习机制 | 会计师的修正回写规则库，MVP 可先只记录、不做复杂在线学习 |
| 付款合规 | 已通过「不碰钱」规避；若未来要代付，需重新评估 MSB/支付牌照 |

## 八、竞品坐标（讨论依据，2026-07 核实）

- **前半段（收单→OCR→分类→录入）**：Dext / Hubdoc（随 Xero 免费）/ Tofu（gotofu.com，$79/月起，主打学习记账习惯自动分类）已成熟——我们靠**联盟渠道 + 华人多语种 + 报税打通 + 人工兜底**做出区隔，而非在识别技术上硬拼。
- **后半段（审批→付款→归档）**：Bill.com / Plooto / Tipalti 的 AP 自动化地盘，且都**碰钱**、都**非 M365 原生、都不做税**——我们用「不碰钱 + 报税打通」错位竞争。
