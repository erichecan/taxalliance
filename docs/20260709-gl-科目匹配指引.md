# GL 科目匹配指引 —— 每条明细如何落到会计科目

> 日期：2026-07-09
> 面向：理解 AP 工作台里"AI 分类"到底怎么把一条交易匹配到 GL 科目、现状如何、怎么提升准确率。
> 对应代码：[classify-line.ts](../apps/web/src/lib/providers/classify-line.ts)（编排）、[rule-matcher.ts](../apps/web/src/lib/providers/rule-matcher.ts)、[classifier-claude.ts](../apps/web/src/lib/providers/classifier-claude.ts)、[ocr-veryfi.ts](../apps/web/src/lib/providers/ocr-veryfi.ts)、[ingest.ts](../apps/web/src/lib/pipeline/ingest.ts)。

---

## 0. 一句话

每张单据 OCR 出若干**行项目**（描述 + 金额），系统给**每一行**挑一个客户账套里的 **GL 费用科目**，带一个**置信度**；低置信度的交给会计师在复核工作台确认/改正。

---

## 1. 候选科目从哪来（匹配的"靶子"）

匹配的目标集合 = **该客户 QuickBooks 账套的科目表（Chart of Accounts）**。

- 存在本地 `GlAccountCache` 表，`qboAccountId` = **QBO 的 `Account.Id`**（不透明数字，如 `"7"`），这是全系统认的**唯一科目 id**（数据契约 G2）。
- 正式流程是连上客户 QBO 后 `SELECT * FROM Account` 拉回来缓存。**目前是 seed 的 7 个演示科目**（办公用品/电话网络/租金/餐饮/软件订阅/水电/一般费用），真实上线要换成客户 QBO 的真科目表。

> ⚠️ **这是准确率的地基**：靶子不对，再强的模型也没用。比如一张**卖车发票**，如果候选里只有"办公类"科目，就没有"存货/销货成本-车辆"可选，只能勉强归到最接近的。

---

## 2. 匹配的四层优先级（核心机制）

每一行按顺序走这四层，命中即停：

```
一条行项目（描述 + 金额 + 供应商 + 候选科目表）
      │
 ①  规则命中？  ClassificationRule：供应商/关键词 → 科目
      │  命中 → 置信度「高」，来源 rule           ┐
      │  未命中                                    │
 ②  LLM 分类   Claude（有 ANTHROPIC_API_KEY）      │  越靠上
      │  逐行语义判断，从候选科目里选，返回置信度  │  优先级越高
      │  Claude 未配置 → mock（关键词重叠，仅 dev）│
      │  给出科目 → 用它                            │
      │  仍定不了                                   │
 ③  Veryfi 建议  文档级 category（把科目名当       │
      │  categories 传给 Veryfi，它给整单一个建议） │
      │  命中候选科目 → 用它，来源 veryfi           ┘
      │  仍无
 ④  未分类  置信度「低」→ 必须人工
```

**各层细节：**

| 层 | 怎么判 | 置信度 | 何时最有用 |
|----|--------|--------|-----------|
| ① 规则 | 供应商名/描述**包含**规则里的关键词 → 直接给科目 | 高 | 固定供应商（"Bell"→电话网络）、会计师纠正过的 |
| ② Claude | 把"这行 + 候选科目表"给 Claude Haiku，用 tool-use **强制它只能选真实存在的科目 id**，返回科目 + 理由 + 置信度 | 模型判定 | **逐行精准**，一张单里不同行归不同科目 |
| ② mock | 行描述与科目名的**关键词字面重叠**打分（无重叠不硬猜） | 中/低 | 仅 dev 兜底，语义弱 |
| ③ Veryfi | Veryfi 对**整单**给一个你科目表里的建议（如 `Office Supplies` score 0.86） | 由 score 换算 | **单一类目的单据**（一张全是办公用品的收据，一次全归对） |
| ④ 无 | — | 低 | 交人工 |

> 关键区别：**Claude 是逐行的、Veryfi 是文档级的（整单一个）**。所以多类目发票（如 55 行的车行发票）必须靠 Claude 逐行；Veryfi 只作"Claude 也定不了时"的兜底。

---

## 3. 置信度分级 → 决定要不要人工

| 置信度 | 含义 | 处理 |
|--------|------|------|
| 高 | 规则命中，或模型很有把握 | 自动通过（仍可复核） |
| 中 | AI 有建议但不笃定 | 建议 + 人工确认 |
| 低 | 定不了 | **必须人工** |

- 复核工作台**低置信度行自动置顶高亮**。
- 文档级置信度 = 所有行里**最差**的那个（派生，不单独存，契约 §4.7）。
- 全部行都分好类，"确认并录入 QBO"按钮才解锁。

## 4. 现状 vs 目标（诚实对照）

| 维度 | 现在（demo） | 目标（上线） |
|------|-------------|-------------|
| 逐行分类引擎 | **mock**（关键词，语义弱）+ Veryfi 文档级兜底 | **Claude 逐行**（配 `ANTHROPIC_API_KEY`），Veryfi 只兜漏 |
| 候选科目表 | seed 的 7 个演示科目 | 客户 **QBO 真实科目表**（`GlAccountCache` 从 QBO 同步） |
| 规则学习 | 有规则表、能命中 | 会计师改一次 → **自动 upsert 规则**（纠正落库那步还没接） |

> 实测：一张零食收据在**只有办公类演示科目**下，mock 全部匹配不上（0/10），靠 Veryfi 文档级兜底后 10/10 都归到"办公用品"——**能落上但很粗**。这正说明地基（科目表）和逐行引擎（Claude）才是准确率的关键。

## 5. 怎么提升匹配率（按杠杆从大到小）

1. **配 `ANTHROPIC_API_KEY` → 开 Claude 逐行分类**（最大杠杆）
   - `apps/web/.env` 里填上，重启即自动从 mock 切到 Claude。逐行语义判断，多类目单据也能分对。
2. **接客户真实 QBO 科目表**（第二大）
   - 连 QBO 后同步 `Account` 到 `GlAccountCache`。靶子对了，模型才有正确科目可选。
3. **积累规则（学习飞轮）**
   - 会计师每纠正一次 → 写一条 `ClassificationRule`（供应商/关键词 → 科目）→ 下次同类**直接高置信命中**，越用越准、需人工的比例越低。
4. **给 Claude 更多上下文**（进阶）
   - 科目带 `accountType`/`accountSubType`、供应商历史、行业模板，能进一步提准。
5. **描述规整**（已做）
   - 用 Veryfi 的 `normalized_description`（`ENRGY DRNK`→`ENERGY DRINK`），对分类和展示都更友好。

## 6. 会计师在界面怎么控制

复核工作台中栏，每行一个 **GL 科目下拉**：

1. 看识别值 + AI 建议科目 + 置信度色标（低的置顶红）。
2. 下拉里选/改科目 → 右栏"将写入 QBO 的 Bill"**实时更新**。
3. 全部行选好 → 「确认并录入 QBO」。
4. 录入时每行的 `glAccountId` 就是 QBO Bill 行的 `AccountRef.value`。

> **学习机制（待接）**：会计师的每次改正应 upsert 成 `ClassificationRule`，让系统"记住"。这一步（复核落库 + 规则回写）还没接，是紧接的下一个增量。

---

## 7. 数据契约要点（避免各自为政）

- **科目 id 只认 QBO `Account.Id`**（契约 G2）——不要在别处造第二套科目编码。
- 行上的 `glAccountName` 是**下单时冻结的快照**（契约 §4.4），权威名在 `GlAccountCache`（其权威又是 QBO）。
- **报税用的 GIFI 是科目的属性**（`GlAccountCache.gifiCode`），由科目派生，**不另起一套 GIFI 规则表**（契约 §4.5）——记账分类和报税映射共用同一次分类结果。

## 8. 一图流

```
QBO 科目表 ──同步──▶ GlAccountCache（候选靶子，id=QBO Account.Id）
                              │
上传单据 ─OCR─▶ 行项目 ──┬──▶ ① 规则命中？ → 高
                          │     ② Claude 逐行 → 模型置信度
                          │     ③ Veryfi 文档级兜底 → score 换算
                          │     ④ 都不行 → 低/人工
                          ▼
                  复核工作台（改/确认，低置信置顶）
                          │
                  会计师纠正 ──▶（待接）回写 ClassificationRule ──▶ 下次更准
                          ▼
                  确认 → 录入 QBO Bill（行 AccountRef = glAccountId）
```
